# Phase 07: Database Performance Optimization Plan

<objective>
Optimize the Supabase/PostgreSQL data layer to ensure high-speed performance as the lead database scales to hundreds of thousands of records.
</objective>

<tasks>

### Task 1: Advanced Indexing Strategy
- **Composite Indexes**: Add indexes on `(user_id, batch_id, status)` and `(user_id, quality)` to speed up dashboard filtering.
- **JSONB GIN Indexing**: Implement a GIN index on `leads.ai_copy` to allow for lightning-fast searches within AI-generated content.
- **Full-Text Search**: Implement PostgreSQL tsvector indexing on `leads.name` and `leads.address` for global search functionality.

### Task 2: Query & API Refactoring
- **Cursor-based Pagination**: Refactor `api/leads` to use limit/offset or cursor-based pagination to prevent large payload bottlenecks.
- **Server-side Filtering**: Ensure all filtering logic happens at the database level (Postgrest/SQL) rather than in-memory in the Next.js API.
- **View Materialization**: Create a materialized view for global analytics (niche performance) to avoid expensive aggregate joins on every dashboard load.

### Task 3: Connection & Caching Layer
- **Connection Pooling Audit**: Verify Supabase connection pool settings (Transaction vs Session mode) for the FastAPI worker.
- **Redis Integration (Optional)**: Evaluate adding a Redis layer for caching frequent "System Status" and "Batch Metadata" queries.

</tasks>

<success_criteria>
- Query execution plans (EXPLAIN ANALYZE) show index usage for all common dashboard filters.
- API response times for lead fetching remain stable as data volume grows.
</success_criteria>
